## Goal

Change `GET /api/quick-idea-examples` so quick idea examples are generated by
the local Gemma/OpenAI-compatible AI boundary instead of assembled only from
deterministic templates.

## Non-Goals

- Do not add a new third-party AI provider.
- Do not install, download, or tune a local model automatically.
- Do not change the frontend quick-example UI unless the API contract requires it.
- Do not make routine automated tests require a running local model.

## Assumptions

- Slow responses are acceptable for this endpoint when AI generation is available.
- The existing local Gemma/OpenAI-compatible endpoint remains the approved AI boundary.
- If the model is unavailable, the API should still return a valid response through
  a documented fallback so local verification and basic product availability remain
  intact.

## Constraints

- Routes stay thin; model HTTP logic belongs in integration modules.
- AI output must be parsed and validated before entering response models.
- External adapter calls must have explicit timeout configuration.
- Tests must cover AI success and fallback without requiring a live model.

## Outcome

`GET /api/quick-idea-examples` now selects requested Q5 fields and asks local
Gemma4/OpenAI-compatible chat completion to generate final Korean idea example
text. AI output is validated for exact requested fields before entering the
response model. When Gemma is unavailable, times out, or returns invalid JSON,
the endpoint returns deterministic fallback examples so routine verification and
basic availability remain intact.

## Rollback Strategy

Revert the adapter wiring and docs changes to restore deterministic random
quick-example generation.
